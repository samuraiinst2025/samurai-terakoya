👋 Section 0: はじめに

本書は、プログラミング初学者の段階を卒業しつつある方や、講師、非エンジニアのプロジェクトマネージャー（PM）、そして「AIを使って開発を効率的に進めたい実務初心者」に向けたガイドです。

「要件定義」と聞くと、難しそうな分厚いドキュメントを想像するかもしれません。しかし、AIと協働する時代において、その形は大きく変わりつつあります。本書は、初学者向けの「ざっくりした説明」よりも一歩踏み込み、開発者向けの「専門的すぎる仕様書」よりも噛み砕いた、ちょうど中間（実践で使えるレベル）の知識をまとめたものです。

🏗️ Section 1: 従来の要件定義とは？

まずは、AIが登場する前の「従来の要件定義」がどのようなものだったかをおさらいしましょう。

🎯 要件定義の目的

プロジェクトマネジメントの世界標準であるPMI（Project Management Institute）によれば、ビジネス分析の基本は「ニーズを特定し、解決策を推奨し、要求を引き出し、文書化し、管理すること」です。つまり、「誰のどんな課題を、どんなシステムで解決するのかを明確にし、合意すること」が目的です。

📄 国際標準が定める主要な成果物

ソフトウェア要件の国際規格（ISO/IEC/IEEE 29148）では、開発の段階に応じて以下のような文書を作成することが推奨されていました。

  • BRS（ビジネス要求仕様）: なぜこのプロジェクトをやるのか（ビジネス目的）
  • StRS（ステークホルダー要求仕様）: ユーザーや関係者がシステムに何を求めているか
  • SyRS（システム要求仕様）: システム全体として何ができなければならないか
  • SRS（ソフトウェア要求仕様）: ソフトウェアが持つべき具体的な機能や振る舞い

🗂️ 要件の4分類（IIBA / BABOK）

ビジネスアナリシスの知識体系であるBABOKでは、要件を4つに分類しています。

  1. ビジネス要件: 組織としての目標や解決したい問題。
  2. ステークホルダー要件: ユーザーが特定の作業を行うためのニーズ。
  3. ソリューション要件: システムが持つべき機能（機能要件）と、性能やセキュリティなどの品質（非機能要件）。
  4. 移行要件: 旧システムから新システムへ移行するための一時的な要件（データ移行など）。

📊 現場でよく作られていた成果物

これらの規格をベースに、日本の開発現場では以下のような図や表が大量に作られていました。

  • ユースケース図: 誰がシステムで何をするのかを表す図
  • ER図: データベースの構造と関係性を表す図
  • 画面遷移図: 画面から画面への繋がりを表す図
  • 機能一覧 / 非機能要件定義書
  • RTM（要件トレーサビリティマトリクス）: 要件が設計・実装・テストまで漏れなく紐づいているかを追跡する表

従来の流れは、「要件定義（書類作成） → 設計 → 実装（コーディング） → テスト」という、後戻りが難しいウォーターフォール型が主流でした。

⚡ Section 2: バイブコーディング時代の変化

生成AIの登場により、要件定義から実装へのアプローチが根本から覆りました。

🎸 「vibe coding（バイブコーディング）」の誕生

2025年、元OpenAI/TeslaのAndrej Karpathy氏が「vibe coding」という言葉を提唱しました。これは、「雰囲気に身を任せ、コードの存在すら忘れてAIに開発を委ねる」という新しいコーディングスタイルです。

長所と短所
✅ 長所: アイデアからプロトタイプ（試作品）を作るスピードが圧倒的に速い。学習コストが低い。
❌ 短所: 既存の複雑なコードベースや、バグが許されない本番開発（ミッションクリティカルなシステム）においては、AIが文脈を見失いやすく不安定になる。

⚙️ Spec-Driven Development（仕様駆動開発）への進化

バイブコーディングの「とりあえず動くものを作るが、後で破綻する」という弱点を克服するため、GitHubやMicrosoftは「Spec-Driven Development（仕様駆動開発）」を提唱しています。

これは、AIにコードを書かせる前に、しっかりとした「仕様（Spec）」を共有コンテキストとして与えるアプローチです。GitHub Spec Kitでは、以下の3ステップを踏みます。

  • /specify : 「何を・なぜ作るのか」をAIと定義する
  • /plan : 「どうやって作るか（技術スタックやアーキテクチャ）」を計画する
  • /tasks : 「具体的な実装タスク」に分解する

💡 重要な転換: 仕様書は「人間が読むための静的な文書」から、「AIが読んで理解し、実行するための生きた成果物（living artifact）」へと役割を変えました。また、「ドキュメントを完璧にしてからコードを書く」のではなく、「コードや対話からドキュメントを育てる」という順序の逆転が起きています。

⚖️ Section 3: 何が変わり、何が変わらないのか

AI時代においても、「要件定義そのもの」が不要になったわけではありません。情報としての価値は残ったまま、「成果物の形」や「作るタイミング」が変わったのです。

🔴 不要になった（または後ろ倒しになった）もの

手作業で時間をかけて先回りして作るドキュメントは不要になりました。

  • 開発前に人間が手書きで綺麗に整える図面（ER図、ユースケース図、画面遷移図など）
  • 開発を開始する前に「完璧に仕上げる」大部な要求仕様書
🟡 タイミングが変わったもの（リバースエンジニアリング）

設計図を「先に書く」のではなく、AIが書いたコードから「後で逆生成」します。

  • ER図: データベース定義のコード（スキーマ）から自動生成する
  • ユースケース図: AIとの対話や実装結果から、Mermaid形式などで出力させる
  • 詳細仕様書: プロトタイプが動いた後に、現状の仕様をAIに整理させる
🟢 むしろ重要になったもの（AIへのガードレール）

AIが勝手な解釈で暴走しないよう、枠組みやルールを明示することが極めて重要です。

  • 目的（WHY）と対象ユーザー（WHO）の言語化
  • 非機能要件・セキュリティ制約（AIは指示されないとセキュリティを疎かにしがち）
  • スコープ外の明示（「今回は〇〇の機能は作らない」という制限）
  • 受入基準（Done条件）（何をもって完成とするか）
  • AI向け指示ファイル（CLAUDE.md や Copilot custom instructions）

🛤️ Section 4: AI時代の実践フロー

AIを使った開発では、以下のようなサイクルで進めるのが効率的です。

1
目的・ユーザー・制約を整理する（AIが迷わないためのGPS設定）
2
AIと会話しながら試作する（vibe codingで素早く動くものを作る）
3
動くものを確認・修正する（要件のズレをここで直す）
4
コードからドキュメントを逆生成する（ER図や仕様書をAIに書かせる）
5
必要に応じて仕様を整理・共有する（チームや引き継ぎのために保存）

🛠️ Section 5: AIで効率的に成果物を作る方法（実践Tips）

具体的にAIを使ってどのように成果物を作っていくのか、いくつかのテクニックを紹介します。

1. 永続コンテキスト（プロジェクト憲法）を持たせる

AIエディタを利用する際、毎回同じ指示をするのは非効率です。プロジェクトのルートディレクトリに指示ファイルを置くことで、AIの行動を制御します。

  • ファイル名: CLAUDE.md (Claude Code用) や、.github/copilot-instructions.md (GitHub Copilot用)
  • 書く内容: プロジェクトの目的、ビルド・テストの実行コマンド、アーキテクチャの規約、命名規則、やってはいけない禁止事項。
  • 効果: AIがプロジェクトの「文脈」を常に記憶し、ルールを守ってコードを生成するようになります。

2. Mermaid記法で図面を「逆生成」する

手書きで図形を配置するのではなく、AIにテキストデータ（Mermaid形式）で図を書かせます。

プロンプト例：
「現在の schema.prisma（データベース定義）の構造を読み取って、データベースのER図をMermaid形式で出力してください。テーブル間のリレーションシップも明記してください。」

※出力されたコードは、Mermaid Live Editor に貼り付けるだけで、綺麗な図として表示されます。

3. GitHub Spec Kitで仕様を共同作成する

仕様をゼロから書くのではなく、ツール（CLI）を使ってAIと対話しながら作り上げます。/specifyで要件を固め、/planで技術選定を行い、/tasksで実装手順に落とし込むことで、強固な基盤を作ることができます。

4. 受入基準を「Gherkin形式」で自動生成する

言葉の曖昧さをなくすため、テストしやすい形式（Given / When / Then）に変換させます。

プロンプト例：
「今回作成したログイン機能の受入基準（テストシナリオ）を、Gherkin形式（Given/When/Then）で網羅的に書き出してください。正常系だけでなく、パスワード間違いなどの異常系も含めてください。」

✅ Section 6: AI時代の要件整理チェックリスト

AIに開発を依頼する前に、以下の「5つの問い」に対する答えを用意しておきましょう。これが、AIへの最高のプロンプト（指示書）になります。

  • ✅ WHY（目的）: なぜこれを作るのか？作ることで、誰のどんな悩みが解決するのか？（背景とゴール）
  • ✅ WHO（ユーザー）: 使う人は誰か？ 管理者と一般ユーザーなど、複数の役割がある場合、それぞれ何ができるのか？
  • ✅ LIMIT（制約）: いつまでに作るか？ 必須で使う技術やツールはあるか？ セキュリティ上の制限は何か？
  • ✅ DONE（完了基準）: どのような状態になれば「完成」と言えるのか？ どのようにテストして確認するのか？
  • ✅ OUT（スコープ外）: 今回「やらないこと」は何か？（AIが気を利かせて勝手に余計な機能を作らないための防波堤）
