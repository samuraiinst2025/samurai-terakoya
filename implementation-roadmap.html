🚀 MVP SaaS 化 実装ロードマップ

AI 協働による段階的な SaaS 化実装ガイド

📖 このドキュメントについて

このロードマップは、Firebase ベースの予算管理システムを MVP（Phase 1-3）として SaaS 化する際の実装ガイド です。

各タスクに対して 「AI に相談する観点」 を明記しており、 AI（Claude）と対話しながら実装を進める際の思考枠組みを提供します。

📌 MVP（Phase 1-3）の位置づけ： 「信頼できるマルチテナント基盤」の構築に集中。Claude SDK 統合（AI 機能強化）は Phase 4 以降に分離し、 MVP 提供開始後、ユーザーフィードバックを踏まえて計画的に進めます。

💡 使い方： 各タスクの「観点」を参考に、AI に質問・相談し、具体的な実装方法を掘り下げていってください。

🎯 構築のポイント～MVP で何を実現するのか～

以下の3つのポイントを理解することで、各タスクの意味・目的が明確になります。 Phase 1～3 のタスクは、これらのポイントをクリアするために設計されています。

💡 注記： Claude SDK による AI 機能統合は MVP 外とし、Phase 4 以降に分離します。MVP は「信頼できるマルチテナント基盤」の構築に集中します。

What 1：MVP SaaS化とは何か？

「最小限の機能・信頼ベースで、顧客を一気に獲得する戦略」

  • 無料提供： 5～10社の既存信頼顧客に無料で提供し、市場検証を行う
  • 最小限機能： 既存システムの機能をそのまま活かし、追加開発は最小限にする
  • フィードバック重視： ユーザーからのご意見を集め、次の段階での改善に活かす
  • スピード重視： 2～3ヶ月で本提供開始。完璧さより「市場への迅速な投入」を優先する

→ Phase 1～3 全体を通じて、このポイントをクリアするための準備と実装を進めます。

What 2：マルチテナント化とは何か？

「複数の顧客が同じシステムを使えるように、データとアクセス権を分離すること」

  • テナント ID： 各顧客（テナント）に一意の ID を付与し、識別する
  • データ分離： Firestore のスキーマを修正し、テナント A のデータがテナント B に見えないようにする
  • 権限管理： Firebase Auth とカスタムクレイムで「このユーザーはこのテナントのこのデータを見られる」を制御する
  • セキュリティルール： Firestore ルールで「他のテナントのデータへのアクセスは許可しない」を自動的に実装する

→ Task 1.2（マルチテナント設計）→ Task 2.2（実装）で実現します。

What 3：データセキュリティ・テナント間分離とは何か？

「複数顧客が同じデータベースを使っても、他社のデータが見えない仕組みを保証すること」

  • 完全分離： テナント A のユーザーが、テナント B の予算データを「理論的に」見られない設計にする
  • Firestore ルール： データベースレベルでの強制 - ルール違反のアクセスは自動で拒否される
  • API 層の保護： バックエンド API でも、リクエストしたユーザーの権限を確認し、不正アクセスを防ぐ
  • 監査ログ： 「誰が・いつ・何を・どのデータを」操作したかを記録し、セキュリティインシデント時に調査可能にする
  • なぜ重要？： 顧客が「自分たちの財務データが安全か」を信頼できるかどうかが、SaaS ビジネスの成功を左右する

→ Task 1.2（設計）→ Task 2.2（実装）→ Task 2.6（監査ログ）→ Task 3.1（セキュリティテスト）で実現します。

📋 Phase 1: 準備・設計（Week 1-3）

Task 1.1: 現状分析と設計

目的

既存の Firebase ベースシステムの全体像を理解し、SaaS 化に向けた基礎設計を行う。 Firestore スキーマ、React コンポーネント、Node.js API、既存 AI 機能を把握する。

AI に相談する観点

観点1：現状把握 既存システムの構成要素は何か？（Firestore のコレクション、API エンドポイント、React の主要コンポーネント）
観点2：既存 AI 機能の理解 現在実装されている「入力補助」の AI 機能は具体的に何をしているか？どのデータを使っているか？（どの AI API / サービスを使用しているか）
観点3：アーキテクチャの課題 現在のシステムで、SaaS 化（複数顧客対応）する上での主な課題は何か？
観点4：性能・スケーラビリティ 現在のシステムで 10 社～100 社の規模に対応する場合、どこがボトルネックになるか？

期待される成果物

📄 成果物 「System Current State Analysis Document」
  • Firestore スキーマ図（主要コレクション）
  • API エンドポイント一覧
  • React コンポーネント構成図
  • 既存 AI 機能の詳細（入力補助の動作フロー）
  • SaaS 化に向けた課題リスト
⚠️ 注記 まだ完璧な設計を目指す必要はありません。「現状を理解する」ことが目的です。
Task 1.2: マルチテナント設計

目的

複数の顧客（テナント）が同じシステムを使える仕組みの基本設計を行う。 データ分離、認証・認可、権限管理の戦略を決定する。

AI に相談する観点

観点1：テナント ID 戦略 テナント（顧客）をどう識別・管理するか？（テナント ID の形式、どこに保存するか）
観点2：データ分離の仕組み Firestore でテナント A とテナント B のデータを完全に分離する方法は？（スキーマ設計）
観点3：認証・認可 Firebase Auth で複数テナントに対応する場合、どのようにユーザーを管理し、テナント情報を保持するか？
観点4：シンプルな権限管理 MVP 段階でのシンプルな権限モデル（例：管理者、ユーザー）をどう実装するか？
観点5：Firestore セキュリティルール Firestore ルールで「このユーザーは自分のテナントのデータだけを見る」を実現する方法は？

期待される成果物

📄 成果物 「Multitenancy Design Document」
  • テナント ID 戦略
  • 修正後の Firestore スキーマ（テナント ID 追加）
  • データ分離図
  • Firebase Auth の custom claims 設計
  • Firestore セキュリティルール案
  • 簡易権限モデル（管理者 / ユーザー）
⚠️ 注記 「100% 完璧な設計」を目指さず、「実装可能で、後から修正できる設計」を意識してください。

🔐 マルチテナント化の実装フロー

💡 ポイント： 各テナントのユーザーがシステムにアクセスすると、Firebase Auth が「このユーザーは tenant_001」と認識。 API層とFirestore のセキュリティルールが協力して、自分のテナントのデータだけを返す仕組みです。

Task 2.2: マルチテナント化実装

目的

Firestore スキーマを修正し、複数の顧客がデータを共有することなく使用できる仕組みを実装する。 Firebase Auth と Firestore セキュリティルールで、テナント間のデータ完全分離を実現する。 既存の予算管理機能（予算作成・編集・削除・レポートなど）を、マルチテナント対応に適応させる。

AI に相談する観点

観点1：データ分離戦略 Firestore のコレクション構造をどう修正するか？（各コレクションにテナント ID を追加？新しいコレクション構造？）
観点2：権限管理の実装 Firebase Auth の custom claims で「ユーザーがどのテナントに属するか」をどう管理するか？
観点3：セキュリティルール Firestore ルールで「このユーザーは自分のテナントのデータだけを読み書きできる」ロジックをどう実装するか？
観点4：API 層の修正 Node.js のバックエンド API で、テナント分離をどのレベルで実装するか？（リクエスト段階？クエリ段階？）
観点5：既存データの移行 既存データ（テナント ID がない）を新しいスキーマに移行する方法は？ダウンタイムをどう最小化するか？
観点6：フロントエンドの修正 React で「現在のユーザーが属するテナント」を把握し、UI で正しくデータを表示する実装方法は？

期待される成果物

📄 成果物 「Multitenancy Implementation」
  • 修正後の Firestore スキーマ（実装版）
  • Firebase セキュリティルール（テスト済み）
  • データ移行スクリプト
  • バックエンド API の修正内容
  • フロントエンドの修正内容
  • テスト結果（複数テナント間でデータが分離されていることを確認）
⚠️ 注記 実装中に「テナント A のデータが見えない」「テナント B のクエリが遅い」といった問題が出た場合、即座に AI に相談して解決策を探ってください。

🗂️ Firestore スキーマ設計（マルチテナント対応）

重要： すべてのドキュメントに tenant_id フィールドを追加

💡 tenant_id の役割：
• Firestore のセキュリティルール内で「この tenant_id のデータだけ」と指定できる
• API 層でもこの値を使ってバリデーション「リクエストユーザーの tenant_id とDBの tenant_id が同じか」
• インデックス作成時に「tenant_id + category」「tenant_id + created_at」など、テナント + 属性でクエリを最適化

観点3：パスワードリセット パスワード忘れ時の リセットフロー（メール送信、リセットリンク、新パスワード設定）をどう実装するか？
観点4：セッション管理 ブラウザ側での認証トークン（ID token）の管理、更新、ログアウト処理はどう実装するか？

Task 3.1: テスト・検証

目的

MVP の品質を確保するため、総合テスト（E2E テスト）、セキュリティテスト、負荷テストを実施する。 本番環境に近い環境で、5～10 の並行ユーザーを想定したテストを行う。

AI に相談する観点

観点1：E2E テストシナリオ 「新しいユーザーがテナントに参加する」「予算データを作成・編集する」「AI 入力補助を使う」といった一連の操作をテストする方法は？
観点2：マルチテナント検証 複数のテナント・複数のユーザーでのテストを実施する際、データ分離が本当に機能しているかをどう確認するか？
観点3：セキュリティテスト 「権限のないユーザーが他のテナントのデータを見られないか」「CSRF 攻撃への対抗は大丈夫か」などをどう検証するか？
観点4：負荷テスト 「5～10 ユーザーが同時にシステムを使用する場合のパフォーマンスは？」「データベースのクエリは遅くならない？」をどう測定するか？
観点5：ブラウザ互換性 Chrome、Safari、Firefox などの主要ブラウザでの動作確認は？

期待される成果物

📄 成果物 「Test Completion Report」
  • E2E テスト実行結果（チェックリスト）
  • マルチテナント検証結果
  • セキュリティテスト結果（脆弱性なし確認）
  • 負荷テスト結果（応答時間、エラー率）
  • ブラウザ互換性テスト結果
  • 見つかった問題リスト（優先度付き）
  • 改善作業の記録
⚠️ 注記 すべてのテストを「100% 完璧に」こなす必要はありません。「本番環境に悪影響を与えない程度の品質」を目安にしてください。
Task 3.2: ユーザーオンボーディング

目的

初期ユーザーに対して、システムの利用開始をサポートする。 招待メール送信、アカウント作成ガイド、初期セットアップ（テナント情報登録、Freee 連携など）を実施する。

AI に相談する観点

観点1：招待メール 初期ユーザーへの招待メールには、どのような情報・リンクを含めるべきか？
観点2：オンボーディング資料 「初日で何をすべきか」を分かりやすく説明する資料（動画？ステップガイド？）をどう提供するか？
観点3：Freee 連携セットアップ ユーザーが自分で Freee API キーを取得・設定できるようなガイドが必要か？
観点4：初期データ入力 最初の予算データを入力するまでのサポート（テンプレート提供？入力例？）をどうするか？
観点5：初期サポート 「よくある質問」「よくあるエラー」について、迅速に対応できる体制が必要か？

期待される成果物

📄 成果物
  • 招待メールテンプレート
  • オンボーディングガイド（PDF またはウェブページ）
  • Freee 連携セットアップガイド
  • 初期データ入力テンプレート＆サンプル
  • よくある質問（FAQ）と回答
  • サポート連絡先（メール、Slack など）
⚠️ 注記 オンボーディングは「最高のドキュメント」ではなく、「ユーザーが実際に使いながら分かる」形を目指してください。
Task 3.3: フィードバック収集システム

目的

初期ユーザーから効果的にフィードバックを収集し、次のフェーズでの改善に活かす仕組みを構築する。 「何が良かったか」「何が困ったか」「次に欲しい機能は何か」を体系的に収集する。

AI に相談する観点

観点1：フィードバック収集手段 インアプリアンケート、メール、Slack、定期インタビュー、使用ログ分析など、どの方法を組み合わせるか？
観点2：アンケート設計 「使いやすさ」「AI 入力補助の質」「Freee 連携」「サポート」などの観点で、何を聞くか？
観点3：NPS 測定 「このサービスを友人に勧めるか？」という NPS（Net Promoter Score）をどうやって測定するか？
観点4：フィードバック管理 収集したフィードバックをどのように整理・優先順位付けして、改善に活かすか？
観点5：ユーザーとのコミュニケーション 「ご意見ありがとうございます。このように改善しました」という内容を、ユーザーにどう伝えるか？

期待される成果物

📄 成果物
  • インアプリアンケートフォーム
  • NPS 測定用アンケート
  • フィードバック管理スプレッドシート（テンプレート）
  • 定期インタビュー用ガイド
  • 使用ログから得られたインサイト（例：「AI 入力補助は○○の場面で よく使われている」）
  • フィードバック分析レポート（最初の 2～4 週間分）
⚠️ 注記 フィードバック収集は「継続的」であり、Phase 3 中だけで終わりません。仕組みを整備した後は、定期的に（毎週または隔週）収集・分析を行ってください。
Task 3.4: サポート体制構築

目的

初期ユーザーからのサポート依頼に対応できる体制を整備する。 メール対応、FAQ 拡充、トラブルシューティングの仕組みを構築する。

AI に相談する観点

観点1：サポートチャネル メール、Slack、Zoomなど、どのチャネルでサポートを提供するか？各チャネルの対応時間は？
観点2：サポート SOP お問い合わせを受けたときの対応フロー（受け付け → 確認 → 回答 → 解決確認）をどう標準化するか？
観点3：よくあるトラブル 「ログインできない」「データが同期されない」「AI 入力補助が遅い」といったトラブルの対処方法は？
観点4：FAQ 拡充 初期ユーザーからのご質問をもとに、FAQ をどのように充実させていくか？
観点5：内部コミュニケーション サポート対応の記録・共有（チーム内で「あのユーザーはこういう問題があった」と情報共有）をどのように行うか？

期待される成果物

📄 成果物
  • サポート SOP（標準対応手順書）
  • サポート連絡先と対応時間
  • よくあるトラブル&対応方法（ドキュメント）
  • サポート対応ログ（スプレッドシートまたは専用ツール）
  • FAQ の拡充版
  • チーム内の情報共有ルール
⚠️ 注記 MVP 段階では「完璧なサポート」は目指さず、「迅速・丁寧な対応」を心がけてください。ユーザーの信頼を得ることが、次の段階での成功につながります。

🤖 Phase 4: AI 機能強化 - Claude SDK統合（Month 6～8）

目標：マルチエージェント構成の実装、AI 分析機能の本格化

Phase 1～3 で構築した「信頼できるマルチテナント基盤」の上に、 Claude SDK を活用したマルチエージェント AI 機能を追加します。

親エージェント（Orchestrator）が複数の子エージェント（入力補助、分析）を統合管理し、 ユーザーのリクエストに応じて適切な AI 機能を提供する高度なシステムへと進化します。

Phase 4 で実現すること

  • 🤖 マルチエージェント構成 - 親エージェント（Orchestrator）と複数の子エージェントが協力して動作
  • 💡 入力補助の高度化 - Claude Haiku による軽量で高速な入力支援
  • 📊 分析機能の実装 - Claude Sonnet による支出パターン分析・予測生成
  • 🔄 SDK 自動管理 - エージェント間通信、タイムアウト処理、リトライが自動化される
  • 📈 スケーラビリティ - 新しいエージェント（レポート生成、予算最適化など）を簡単に追加可能

💡 重要： Phase 4 は「MVP 提供開始後」の段階です。初期ユーザーからのフィードバック、システムの安定性確認を踏まえた上で、計画的に進めてください。

Phase 4 のタスク構成（予定）

以下は Phase 4 で想定されるタスクです。詳細設計は、Phase 3 完了時点でのユーザーフィードバックを踏まえて策定します。

  • Task 4.1: Claude SDK 戦略策定 - マルチエージェント構成の詳細設計、API コスト管理、実装優先度決定
  • Task 4.2: 親エージェント（Orchestrator）実装 - ユーザーリクエスト分析、子エージェント選択ロジック
  • Task 4.3: 入力補助エージェント強化 - Claude Haiku 統合、プロンプト最適化、レイテンシ改善
  • Task 4.4: 分析エージェント実装 - Claude Sonnet による支出分析、予測生成、アノマリー検知
  • Task 4.5: エージェント間連携テスト - マルチエージェント動作確認、コスト測定、性能最適化

✅ Phase 4 の成功指標
• マルチエージェント構成が安定動作
• API コスト が予測範囲内
• ユーザーが AI 分析機能を活用できる
• NPS 45 以上（エンタープライズ向け SaaS の目安）
