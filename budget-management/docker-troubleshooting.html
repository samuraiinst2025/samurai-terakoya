🔨 セットアップ段階でのトラブル（Day 1-2）

Docker のインストール・起動

❌ "docker: command not found"

Docker がインストールされていないか、パスが設定されていません。

✅ 解決方法
  • Docker Desktop のインストール確認（Mac/Windows）
  • Linux の場合：apt install docker.io（Debian/Ubuntu）
  • インストール後、ターミナルを再起動
  • docker --version で確認
Docker バージョン確認 docker --version # 出力例：Docker version 20.10.12, build e91ed57

Dockerfile ビルド時のエラー

❌ "docker build: error checking context: 'no such file or directory'"

Dockerfile が見つからないか、ビルドコンテキストが不正です。

✅ 解決方法
  • カレントディレクトリを確認：pwd
  • Dockerfile が存在するか確認：ls -la Dockerfile
  • 正しいパスから実行：docker build -t firebase-budget:latest .
❌ "npm: not found" または "npm ERR! code ENOENT"

Node.js モジュールのインストール時に失敗しています。

✅ 解決方法
Dockerfile の確認 FROM node:18-alpine # ← Alpine なら npm は含まれる # または FROM node:18 # ← フルサイズなら確実に npm が含まれる
  • package.json の確認：ファイルが存在するか、JSON 形式は正しいか
  • COPY package*.json ./ が正しく指定されているか
  • 必要に応じて npm cache clean --force を実行

docker-compose 起動時のエラー

❌ "docker-compose: command not found"

Docker Compose がインストールされていません。

✅ 解決方法
Docker Compose インストール # macOS (Homebrew) brew install docker-compose # Linux sudo curl -L "https://github.com/docker/compose/releases/download/v2.12.0/docker-compose-$(uname -s)-$(uname -m)" \ -o /usr/local/bin/docker-compose sudo chmod +x /usr/local/bin/docker-compose # バージョン確認 docker-compose --version
❌ "ERROR: .env file not found"

.env ファイルが見つかりません。

✅ 解決方法
.env ファイル作成 cp .env.example .env # エディタで開いて、値を入力 nano .env # または vim .env
重要：本番環境での .env
本番環境では .env ファイルを使わず、環境変数を直接設定してください：
export FREEE_API_TOKEN="actual_token_here" export FIREBASE_KEY_PATH="/etc/secrets/firebase-key.json" docker-compose up
❌ "port 5000 is already allocated"

別のプロセスがポート 5000 を使用しています。

✅ 解決方法
ポート確認と解放 # macOS / Linux lsof -i :5000 # プロセスID を確認したら終了 kill -9 <PID> # または docker-compose で別ポートを使用 # docker-compose.yml の ports セクションを変更 # ports: # - "5001:5000" ← ホストポート 5001 に変更

🏃 実行・テスト段階でのトラブル（Day 3-4）

Firebase 接続エラー

❌ "GOOGLE_APPLICATION_CREDENTIALS points to invalid service account"

Firebase サービスアカウントキーが見つからないか、パスが不正です。

✅ 解決方法
  • ファイル存在確認：ls -la /path/to/firebase-key.json
  • docker-compose.yml の volumes セクションを確認
  • パーミッション確認：chmod 400 firebase-key.json
  • JSON 形式が正しいか確認：cat firebase-key.json | jq .
volumes セクションの例 volumes: - /path/to/firebase-key.json:/app/secrets/firebase-key.json:ro # ↑ ローカルパスから コンテナ内パスへ（読み取り専用）
❌ "Error: 3 INVALID_ARGUMENT: could not parse service account"

Firebase キーファイルが破損しているか、形式が不正です。

✅ 解決方法
キーファイル検証 # JSON 形式チェック node -e "console.log(JSON.stringify(require('./firebase-key.json'), null, 2))" # または cat firebase-key.json | jq . # エラーが出た場合、Google Cloud Console から再度ダウンロード

Freee API 接続エラー

❌ "401 Unauthorized" または "Invalid API token"

Freee API トークンが不正です。

✅ 解決方法
  • .env ファイルの FREEE_API_TOKEN が正しいか確認
  • トークンにスペースが含まれていないか確認
  • トークンの有効期限が切れていないか確認（Freee の管理画面）
  • 必要な権限（スコープ）が設定されているか確認
トークン確認 # コンテナ内で環境変数を確認 docker-compose exec firebase-budget-app env | grep FREEE_API_TOKEN

コンテナの起動失敗

❌ "docker-compose up" でコンテナが起動しない

アプリケーション内部でエラーが発生しています。

✅ 解決方法
ログ確認 # コンテナのログを表示 docker-compose logs -f firebase-budget-app # 特定のエラーを検索 docker-compose logs | grep ERROR
  • エラーメッセージをよく読んで原因を特定
  • 必要に応じて環境変数を修正
  • docker-compose up で再度試行

⚡ パフォーマンス問題と最適化

問題：コンテナのメモリ使用量が高い

💡 原因の特定方法
メモリ使用量確認 docker stats firebase-budget-app

解決策

  • ヒープサイズの制限：NODE_OPTIONS="--max-old-space-size=512" を環境変数に設定
  • ガベージコレクション設定の確認
  • 不要な依存パッケージの削除
  • docker-compose.yml のメモリ制限を確認（deploy.resources.limits.memory）
docker-compose.yml でのメモリ制限設定 services: firebase-budget-app: environment: NODE_OPTIONS: "--max-old-space-size=512" deploy: resources: limits: memory: 1G

問題：API レスポンス時間が遅い

Freee API のレート制限

  • Freee API は分単位でレート制限があります（通常 1000 リクエスト/分）
  • ループ処理でレート制限に達する場合は、遅延を追加
レート制限対策（コード例） // API 呼び出し間に遅延を追加 const delay = (ms) => new Promise(resolve => setTimeout(resolve, ms)); for (const company of companies) { await fetchFreeeData(company); await delay(100); // 100ms の遅延 }

クエリの最適化

  • Firestore クエリに インデックス を作成
  • 必要なフィールドのみを取得（projection）
  • キャッシング機構を導入（Redis など、将来的に）

問題：CPU 使用率が高い

デバッグ方法

CPU プロファイリング # Node.js のビルトインプロファイラを使用 node --prof app.js node --prof-process isolate-*.log > profile.txt

対策

  • 無限ループやスレッド化されていない処理がないか確認
  • 定期実行タスク（cron）の実行頻度を確認
  • 複雑な計算処理をワーカースレッドに移す

🏢 本番環境での注意点と対応

❌ "container exits immediately"

本番環境でコンテナが起動直後に終了します。

✅ 解決方法
詳細ログ確認 docker-compose logs --tail=50 firebase-budget-app # または docker logs <container_id>
  • 環境変数の設定を確認
  • Firebase キーファイルのパーミッション確認（chmod 400）
  • ディスク容量確認：df -h
  • メモリ不足がないか確認：free -h

自動再起動の確認

再起動ポリシー設定確認 # docker-compose.yml が restart: always に設定されているか確認 grep "restart:" docker-compose.prod.yml

ヘルスチェック監視

ヘルスチェックステータス確認 docker inspect --format='{{json .State.Health}}' <container_id> | jq . # 結果例： # { # "Status": "healthy", # "FailingStreak": 0, # "Log": [...] # }
  • Status が "healthy" であることを確認
  • "unhealthy" の場合、FailingStreak 回数を確認
  • ログから失敗理由を特定

❓ よくある質問（FAQ）

Q. Docker イメージのサイズが大きい場合、どうすればいい？
A. マルチステージビルド、Alpine ベースイメージ、不要なファイルの除外（.dockerignore）で削減できます。
目標： 300MB 以下に抑える
Q. 開発環境と本番環境で設定ファイルを分ける必要がある？
A. はい、必須です。開発環境では hot-reload、ログ出力などが必要ですが、本番環境では異なります。
推奨： docker-compose.yml と docker-compose.prod.yml を分離
Q. コンテナ内のログをどう管理する？
A. コンテナは stdout/stderr へログを出力し、Docker が管理します。
確認方法： docker logs <container_id>
ローテーション設定： logging.options で max-size, max-file を指定
Q. Docker イメージを更新したい場合、どうする？
A. 新しいイメージをビルド、タグ付けして本番環境でプル・再起動します。
# ローカルビルド docker build -t firebase-budget:v1.0.1 . # 本番環境で更新 docker-compose -f docker-compose.prod.yml pull docker-compose -f docker-compose.prod.yml down docker-compose -f docker-compose.prod.yml up -d
Q. コンテナ内で対話的にコマンドを実行したい場合？
A. docker-compose exec で実行できます。
# シェル開始 docker-compose exec firebase-budget-app sh # 特定コマンド実行 docker-compose exec firebase-budget-app npm list
Q. 本番環境でのバックアップはどうする？
A. Firebase は自動バックアップを提供します。定期的にエクスポート操作を手動実行するか、
Cloud Firestore のバックアップ&リストア機能を使用してください。
参考： Google Cloud ドキュメント「Cloud Firestore のバックアップと復元」
Q. Docker と Kubernetes の関係は？
A. Docker はコンテナ化技術、Kubernetes は複数コンテナをまとめて管理するオーケストレーション・ツールです。
小〜中規模： docker-compose で十分
大規模・マイクロサービス： Kubernetes の検討

問題報告テンプレート

Slack や GitHub Issues で報告する際は、以下情報を含めてください：

  • エラーメッセージ全文
  • docker --version, docker-compose --version の出力
  • docker-compose logs の最後 50 行
  • docker stats の出力（パフォーマンス問題の場合）
  • 実行環境（macOS / Linux / Windows）
  • 再現手順（再現が可能な場合）
