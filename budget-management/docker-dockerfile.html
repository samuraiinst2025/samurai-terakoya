📦 Dockerfile の作成

基本構造と概念

Dockerfile は Docker イメージを構築するための指示書です。
Firebase アプリケーション向けの Dockerfile では、以下を定義します：

  • ベースイメージ：Node.js が含まれた環境
  • 依存パッケージ：npm install で必要なライブラリをインストール
  • ソースコード：アプリケーションファイルをコンテナにコピー
  • エントリーポイント：コンテナ起動時の実行コマンド

最小限の Dockerfile 例

Dockerfile (開発環境向け) FROM node:18-alpine WORKDIR /app # 依存関係ファイルをコピー COPY package*.json ./ # パッケージをインストール RUN npm install # アプリケーションコードをコピー COPY . . # ポート公開 EXPOSE 5000 # アプリケーション起動 CMD ["npm", "run", "dev"]
💡 node:18-alpine を選ぶ理由
Alpine Linux ベースイメージは非常に軽量（約 50MB）で、デプロイが高速です。
フルサイズの node:18 （約 900MB）と比べて、10分の1以下のサイズで済みます。

本番環境向け Dockerfile（マルチステージビルド）

マルチステージビルドを使うことで、開発時のツール（テストフレームワークなど）を除外し、
本番イメージのサイズを最小化できます。

Dockerfile (本番環境向け - マルチステージ) # ビルドステージ FROM node:18-alpine AS builder WORKDIR /app COPY package*.json ./ RUN npm ci --only=production # 本番ステージ FROM node:18-alpine WORKDIR /app # ビルドステージから本番パッケージのみコピー COPY --from=builder /app/node_modules /app/node_modules # ソースコード COPY . . # セキュリティ：非rootユーザーで実行 RUN addgroup -g 1001 -S nodejs RUN adduser -S nodejs -u 1001 USER nodejs EXPOSE 5000 HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \ CMD node -e "require('http').get('http://localhost:5000/health', (r) => {if (r.statusCode !== 200) throw new Error(r.statusCode)})" CMD ["node", "dist/index.js"]
⚠️ セキュリティ上の考慮
本番環境では、root ユーザーではなく非 root ユーザーでアプリケーションを実行すべきです。
これにより、コンテナが侵害された場合のリスクを軽減できます。

.dockerignore ファイル

Docker ビルド時に不要なファイルをコピーしないよう指定します：

.dockerignore node_modules npm-debug.log dist .git .gitignore .env .env.local .env.*.local .DS_Store coverage .vscode .idea *.swp *.swo *~ .nyc_output README.md LICENSE .dockerignore Dockerfile docker-compose*.yml .github .gitlab-ci.yml .circleci

Dockerfile のベストプラクティス

ベストプラクティス 説明
最小限のレイヤー数 RUN コマンドを && でつなぎ、レイヤー数を減らす（イメージサイズ削減）
キャッシュの活用 package.json を先にコピーして npm install 後、ソースコードをコピー
HEALTHCHECK の設定 コンテナの健全性を監視し、異常時に自動再起動
非 root ユーザー セキュリティ上、アプリケーションは非 root で実行
.dockerignore 使用 node_modules や .env など不要なファイルを除外

🔗 docker-compose.yml の作成

docker-compose とは

複数のコンテナをまとめて管理するツールです。
このプロジェクトでは、単一コンテナで十分ですが、
将来的に Postgres や Redis など他のサービスを追加する場合に備えて、
スケーラブルな構成にしています。

開発環境用 docker-compose.yml

docker-compose.yml (開発環境) version: '3.8' services: firebase-budget-app: build: context: . dockerfile: Dockerfile container_name: firebase-budget-dev environment: NODE_ENV: development GOOGLE_APPLICATION_CREDENTIALS: /app/secrets/firebase-key.json FREEE_API_TOKEN: ${FREEE_API_TOKEN} FREEE_API_BASE_URL: https://api.freee.co.jp volumes: - .:/app - /app/node_modules - ${FIREBASE_KEY_PATH}:/app/secrets/firebase-key.json:ro ports: - "5000:5000" - "9229:9229" # Node.js デバッグポート restart: unless-stopped networks: - firebase-budget-network networks: firebase-budget-network: driver: bridge

本番環境用 docker-compose.yml

docker-compose.prod.yml (本番環境) version: '3.8' services: firebase-budget-app: image: your-registry/firebase-budget:v1.0.0 container_name: firebase-budget-prod environment: NODE_ENV: production GOOGLE_APPLICATION_CREDENTIALS: /app/secrets/firebase-key.json FREEE_API_TOKEN: ${FREEE_API_TOKEN} volumes: - /etc/firebase-budget/secrets/firebase-key.json:/app/secrets/firebase-key.json:ro - firebase-budget-logs:/app/logs ports: - "5000:5000" restart: always # リソース制限 deploy: resources: limits: cpus: '0.5' memory: 1G reservations: cpus: '0.25' memory: 512M # ヘルスチェック healthcheck: test: ["CMD", "curl", "-f", "http://localhost:5000/health"] interval: 30s timeout: 10s retries: 3 start_period: 10s # ロギング logging: driver: "json-file" options: max-size: "10m" max-file: "3" networks: - firebase-budget-network volumes: firebase-budget-logs: driver: local networks: firebase-budget-network: driver: bridge

環境変数の管理

docker-compose では .env ファイルから環境変数を読み込みます：

.env (ローカル開発用 - gitignore に追加) # Firebase FIREBASE_KEY_PATH=/path/to/firebase-key.json # Freee API FREEE_API_TOKEN=your_freee_api_token_here # Node 環境 NODE_ENV=development
.env.example (リポジトリに含める) # Firebase - 本番環境では /etc/firebase-budget/secrets に配置 FIREBASE_KEY_PATH=/path/to/firebase-key.json # Freee API - 本番環境では Secret Manager で管理推奨 FREEE_API_TOKEN=your_freee_api_token_here # Node 環境 NODE_ENV=development
💡 本番環境での認証情報管理
開発環境： .env ファイル使用（ローカルのみ）
本番環境： 環境変数を直接設定するか、Secret Manager 使用

理由：.env ファイルが誤ってコミットされるリスクを避けるため

docker-compose コマンド一覧

コマンド 説明
docker-compose up コンテナ起動（フォアグラウンド）
docker-compose up -d コンテナ起動（バックグラウンド）
docker-compose down コンテナ停止・削除
docker-compose logs -f ログ表示（リアルタイム）
docker-compose ps 実行中のコンテナ確認
docker-compose exec firebase-budget-app sh コンテナ内で対話的シェル実行
docker-compose restart コンテナ再起動
docker-compose -f docker-compose.prod.yml up 本番設定ファイルを使用

⚙️ 重要な設定値の解説

environment セクション

変数名 用途 開発環境 本番環境
NODE_ENV 実行環境識別 development production
GOOGLE_APPLICATION_CREDENTIALS Firebase 認証ファイルパス /app/secrets/key.json /app/secrets/firebase-key.json
FREEE_API_TOKEN Freee API トークン ローカルの .env から 環境変数で直接指定

volumes セクション（開発環境）

• .:/app - ローカルコードをコンテナと同期（ホットリロード可能）
• /app/node_modules - node_modules をコンテナのみに保持（ホストと分離）
• ${FIREBASE_KEY_PATH}:/app/secrets/firebase-key.json:ro - Firebase キーをマウント（読み取り専用）

ports セクション

開発環境：
• 5000:5000 - メイン API ポート
• 9229:9229 - Node.js デバッグポート（VSCode デバッグ用）

本番環境：
• 5000:5000 - メイン API ポートのみ公開

restart ポリシー

ポリシー 説明 推奨環境
no 自動再起動なし 開発環境
unless-stopped 停止まで自動再起動（停止後は再起動しない） 開発環境
always 常に自動再起動 本番環境
on-failure エラー時のみ再起動（回数指定可能） 本番環境

リソース制限（本番環境）

limits（上限）：
• cpus: '0.5' - CPU使用率を50%に制限
• memory: 1G - メモリを1GBに制限

reservations（確保）：
• cpus: '0.25' - 最低25%のCPUリソース確保
• memory: 512M - 最低512MBのメモリ確保

これにより、他のコンテナとのリソース競合を制御できます。

ヘルスチェック設定

healthcheck: test: ["CMD", "curl", "-f", "http://localhost:5000/health"] interval: 30s # 30秒ごとにチェック timeout: 10s # 10秒でタイムアウト retries: 3 # 3回失敗で unhealthy と判定 start_period: 10s # 起動から10秒間はチェック対象外

ヘルスチェック成功時は healthy、
失敗時は unhealthy ステータスに変わり、
Docker デーモンが自動的に再起動できます。

🔧 よくあるエラーと対処法

Dockerfile ビルド時のエラー

❌ "npm: not found"

原因： node:18-alpine のベースイメージに npm がインストールされていない
対策： node:18 （フルサイズ）を使用するか、Alpine 向けのセットアップを確認

❌ "COPY failed: file not found"

原因： Dockerfile の COPY コマンドで存在しないファイルを指定している
対策： ファイルパスの確認、.dockerignore の確認

docker-compose 実行時のエラー

❌ "port 5000 is already allocated"

原因： ポート 5000 が既に別のプロセスで使用されている
対策： ports セクションを "5001:5000" に変更するか、既存プロセスを停止

# ポート確認 lsof -i :5000 # プロセス終了 kill -9 <PID>

❌ "FIREBASE_KEY_PATH: no value"

原因： .env ファイルに FIREBASE_KEY_PATH が定義されていない
対策： .env.example を参考に .env ファイルを作成

💡 Docker & docker-compose のベストプラクティス

1. イメージサイズの最小化

  • Alpine ベースイメージを使用（通常イメージの 1/10 以下のサイズ）
  • マルチステージビルドで開発用ツールを除外
  • .dockerignore で不要なファイルを除外

2. セキュリティ対策

  • 非 root ユーザーで実行
  • 機密情報（.env、キーファイル）は Git リポジトリから除外
  • ボリュームマウントで :ro（読み取り専用）を指定
  • 定期的にベースイメージをアップデート

3. ログとモニタリング

  • stdout/stderr へのログ出力を基本とする
  • json-file ログドライバーでローテーション設定
  • HEALTHCHECK で自動復旧機能を実装

4. 開発と本番の分離

  • docker-compose.yml と docker-compose.prod.yml を分離
  • 開発環境：hot reload 有効、volumes マウント
  • 本番環境：リソース制限、自動再起動、ヘルスチェック
