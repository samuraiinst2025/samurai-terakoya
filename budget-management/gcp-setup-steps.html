準備
Phase 0: Terraform 準備・テンプレート作成
Terraform スクリプトを開発し、テンプレート化する（初回のみ）
実施時期
初回のみ
工数
3-4日
🎯 なぜこれが必要か？
Terraform スクリプトを一度作成すると、2社目以降の顧客セットアップが大幅に高速化されます。 手作業で GCP を設定すると、毎回ミスのリスクがありますが、Terraform を使えば 「変数だけ変えて実行」で同じ環境が自動で構築されます。つまり、初回の投資が後々の効率化に繋がります。
STEP 0-1
Terraform ファイル構成設計
2-3日
✓ GCP リソースの整理
Terraform で管理すべき GCP リソース（Cloud Functions、Firestore、IAM ロール等）を確認
✓ Terraform ファイル構造の設計
main.tf（メインリソース）、variables.tf（変数）、outputs.tf（出力）、terraform.tfvars（環境値）の設計
推奨構造 project/ ├── terraform/ │ ├── main.tf # Cloud Functions, Firestore の定義 │ ├── variables.tf # 変数定義 │ ├── outputs.tf # 出力値定義 │ ├── terraform.tfvars # 環境別設定（本番・テスト） │ └── provider.tf # GCP プロバイダー設定 └── docs/ └── TERRAFORM_README.md # 使い方ガイド
STEP 0-2
Terraform コード実装・動作確認
3-4日
✓ main.tf の実装
Firebase リソース（Cloud Functions、Firestore）を Terraform で定義
main.tf 例：Cloud Functions resource "google_cloudfunctions_function" "freee_sync" { name = "freee-budget-sync-${var.environment}" runtime = "nodejs18" trigger_http = true source_archive_bucket = google_storage_bucket.functions_code.name source_archive_object = google_storage_bucket_object.functions_zip.name entry_point = "syncFreeeData" environment_variables = { GOOGLE_APPLICATION_CREDENTIALS = var.firebase_key_path } }
✓ IAM ロール・セキュリティ設定
Cloud Functions 実行サービスアカウント、Firestore アクセス権限を定義
✓ ローカルテスト実行
terraform plan で差分確認、terraform validate で構文チェック
✅ STEP 0 チェックポイント
• Terraform スクリプトがローカルで正常に動作する
• terraform plan で期待通りのリソースが出力される
• 顧客ごとの可変項目（プロジェクト ID、環境名等）が variables.tf に定義されている
• Terraform 実行ドキュメントが整備されている
実装
Phase 1-7: 顧客環境への GCP 構築・テスト
Terraform を使って顧客の GCP アカウントに本番環境を構築・テストする（毎顧客実施）
実施時期
顧客ごと
工数
7日
🎯 なぜこれが必要か？
各顧客が独立した GCP アカウントを持つため、本番環境をセットアップする必要があります。 Terraform を活用することで、テンプレートから自動構築し、検証・調整を効率的に進められます。 この期間で、顧客固有の要件調整やパフォーマンス最適化を行います。
STEP 1
顧客環境準備・Terraform 実行
1-2日
顧客ごと
✓ GCP プロジェクト作成・権限設定
顧客の GCP アカウントで新規プロジェクト作成、Terraform 実行用のサービスアカウント設定
顧客ごと
✓ terraform.tfvars の顧客化
テンプレートの variables.tf に顧客固有の値（プロジェクト ID、環境名、リージョン等）を代入
顧客ごと
✓ terraform apply で自動構築
Terraform コマンドでリソースを自動作成。数分で GCP 環境が整備される
STEP 2-3
統合テスト・調整
2-3日
顧客ごと
✓ Firebase Cloud Functions 動作確認
Freee API からのデータ取得、Firestore への書き込みテスト
顧客ごと
✓ IAM ロール・権限動作確認
サービスアカウントが正しい権限を持っているか、Firestore ルールが機能するか確認
顧客ごと
✓ パフォーマンス・スケーラビリティ検証
大規模データでの動作確認、Cloud Functions のメモリ・タイムアウト設定の調整
STEP 4-5
セキュリティ検証・本番準備
2-3日
顧客ごと
✓ Secret Manager でのキー管理
Freee API トークン、Firebase キーが Secret Manager に安全に保存されているか確認
顧客ごと
✓ Firestore セキュリティルール検証
ルールが正しく適用され、未認可アクセスが拒否されるか確認
顧客ごと
✓ バックアップ・復旧策確認
Firestore バックアップが定期実行され、復旧テストが可能か確認
STEP 6-7
ドキュメント・運用引き継ぎ
2-3日
顧客ごと
✓ セットアップ記録ドキュメント作成
実際に実行した terraform.tfvars の値、カスタマイズ内容を記録
顧客ごと
✓ 運用ガイド・トラブルシューティング作成
顧客向けの日次・週次チェック項目、よくあるエラーと対処法を記載
顧客ごと
✓ 顧客側スタッフ向けトレーニング
Terraform 更新方法、ログ確認方法、問題発生時の連絡フローを説明
✅ STEP 1-7 チェックポイント
• Terraform で自動構築した環境が本番で正常に動作している
• すべての統合テストが成功している
• セキュリティ要件が満たされている
• 顧客向けドキュメント・トレーニングが完了している
• Terraform コードとドキュメントが版管理に登録されている

💡 Terraform 運用のベストプラクティス

バージョン管理

Terraform ファイルは Git で管理し、変更履歴を保持します。

環境の分離

本番と開発で異なる terraform.tfvars を用意し、環境混同を防ぎます。

terraform state の安全管理

terraform state（リソース管理状態）は Git に含めず、Google Cloud Storage で共有・保護します。

CI/CD 統合（将来的）

GitHub Actions 等で自動テスト・デプロイを実装すれば、さらに人的作業を削減できます。
