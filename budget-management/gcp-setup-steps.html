🔧 フェーズ1: Terraform スクリプト作成（Day 1-2）

Day 1
Terraform 基盤構築
1日

✓ Terraform ファイル構成の設計

GCP プロジェクト用の Terraform ファイルの構成を決定します。

  • main.tf - メインリソース定義
  • variables.tf - 変数定義
  • outputs.tf - 出力値定義
  • terraform.tfvars - 環境別設定値
💡 ポイント： モジュール化できるものは early から検討。後の保守が楽になります。

✓ GCP プロバイダー設定

Terraform が GCP と通信するための認証・プロバイダー設定を行います。

  • terraform block で Terraform バージョン指定
  • Google provider の設定
  • サービスアカウント認証情報の参照
💡 ポイント： 認証情報は .tfvars に記載せず、環境変数または別ファイルで管理するのがベストプラクティス。

✓ Firebase プロジェクト作成スクリプト

GCP プロジェクト内に Firebase プロジェクトを自動作成するコード書き始め。

  • google_firebase_project リソース定義
  • プロジェクト ID、名前の変数化
  • リージョン設定（asia-northeast1 推奨）
⚠️ 注意： Firebase プロジェクト作成後は削除が制限されます。テスト環境で十分に検証してから本実装を進めてください。
Day 2
Firestore & IAM 設定
1日

✓ Firestore データベース初期化

Firestore の自動構築と初期スキーマ設定を Terraform で実装します。

  • google_firestore_database リソース作成
  • デフォルトセキュリティルール設定
  • データベース ID の指定
  • テストモード vs 本番モードの検討
💡 ポイント： 初期段階では「テストモード」で作成し、セキュリティルール完成後に本番モードに切り替えるのがお勧め。

✓ Cloud Functions リソース定義

Freee データ取得用の Cloud Functions を定義します。

  • google_cloudfunctions_function リソース
  • ランタイム指定（Node.js 18/20）
  • 環境変数設定（Freee API トークン参照等）
  • トリガー設定（HTTP または Pub/Sub）
💡 ポイント： Freee API トークンなどのシークレット情報は Secret Manager と連携するのが安全です。

✓ IAM ロール & Secret Manager 設定

セキュリティと権限管理の基盤を構築します。

  • Cloud Functions 用の IAM ロール割り当て
  • Firestore へのアクセス権限
  • Secret Manager リソース作成
  • Freee API トークンなどの格納
💡 ポイント： 最小権限の原則を適用し、必要な権限のみを付与することが重要です。

⚙️ フェーズ2: Firebase 設定・デプロイパイプライン（Day 3-4）

Day 3
Firebase Hosting & CI/CD
1日

✓ Firebase Hosting 設定

Web UI（React）をデプロイするための Firebase Hosting を設定します。

  • firebase.json の作成・編集
  • ビルド出力ディレクトリの指定（build/）
  • リライトルール設定（SPA 対応）
  • キャッシュ設定
💡 ポイント： SPA（Single Page Application）の場合、index.html へのリライトルール設定が必須です。

✓ GitHub Actions でのデプロイパイプライン

コミット時に自動でビルド・デプロイされる CI/CD を構築します。

  • .github/workflows/deploy.yml 作成
  • npm build トリガー
  • Firebase CLI でのデプロイ
  • Firebase トークン管理（Secrets）
⚠️ 注意： Firebase トークンは GitHub Secrets に安全に保管し、リポジトリコミットに含めないようにしてください。
Day 4
Cloud Functions 最適化 & Firestore ルール
1日

✓ Cloud Functions の詳細実装

既存の Freee 連携ロジックを Cloud Functions 内で最適化します。

  • Freee API 呼び出しロジックの確認
  • タイムアウト設定（デフォルト 60秒）
  • メモリ設定（256MB ～ 8GB）
  • エラーハンドリング & ログ記録
💡 ポイント： 定期実行の場合は Cloud Scheduler と組み合わせ、定期的にデータを同期させるのが効果的。

✓ Firestore セキュリティルール作成

Firestore へのアクセス制御を細かく設定します。

  • ユーザー認証確認ルール
  • 企業ごとのデータ分離ルール
  • 読み取り・書き込み権限設定
  • データ検証ルール
💡 ポイント： セキュリティルールのテストは Firebase Emulator Suite を使うと便利です。

✓ ロギング & 監視設定

本番環境での問題早期発見のためのログ設定。

  • Cloud Logging の設定
  • エラーアラート設定
  • パフォーマンスメトリクス監視

📋 フェーズ3: ドキュメント・検証（Day 5-7）

Day 5-6
ドキュメント作成
2日

✓ 顧客向けセットアップガイド

顧客が自社で実行できるようにステップバイステップのガイドを作成。

  • 「GCP アカウント作成手順」（スクリーンショット付き）
  • 「新規プロジェクト作成手順」
  • 「Terraform 実行手順」
  • 「Freee API トークン取得・設定方法」
  • 「初回セットアップチェックリスト」
💡 ポイント： スクリーンショットは日本語ガイドなら日本語UI での画面が理想。視覚的な分かりやすさが重要です。

✓ トラブルシューティングドキュメント

よくある質問や エラーへの対応を事前にまとめます。

  • 「Terraform 実行時のエラー」対応
  • 「IAM 権限不足」エラー対応
  • 「Freee API 接続エラー」対応
  • 「Firestore セキュリティルール拒否」エラー対応
  • FAQ セクション

✓ セットアップテンプレート用意

顧客が簡単に利用できるテンプレートファイルを準備。

  • terraform.tfvars テンプレート
  • GCP IAM ロール設定スクリプト
  • Secret Manager 初期化スクリプト
Day 7
検証・最終調整
1日

✓ サンドボックス環境での実装テスト

提供するドキュメントが本当に機能するか、実際に試します。

  • 別の GCP アカウント・プロジェクトで実行テスト
  • ドキュメント通りに実行して引っかかった箇所を修正
  • 所要時間を実測（見積もりとの差異確認）
💡 ポイント： 可能なら社内の技術者だけでなく、非技術者にも実行してもらい、フィードバックを得ると良いです。

✓ パフォーマンス・セキュリティチェック

本番環境での安定性を確認。

  • Cloud Functions の実行時間測定
  • Firestore クエリのインデックス最適化
  • セキュリティ設定の再確認
  • コスト予測の算出

✓ デプロイ承認 & リリース

すべての準備が整い、リリース可能状態に。

  • ドキュメント最終レビュー
  • 社内チェックリスト確認
  • GitHub にコード・ドキュメントをプッシュ
  • 顧客向けリリース準備完了
