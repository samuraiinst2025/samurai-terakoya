1

Dockerfile 作成 & ローカルテスト

Firebase プロジェクト構造の確認と Docker 環境の基礎構築

8-10時間
1
既存 Firebase プロジェクトの構造確認
• package.json, firebase.json, .env ファイルの確認
• 依存関係（npm packages）の整理
• 環境変数（GOOGLE_APPLICATION_CREDENTIALS等）の抽出
2
Dockerfile 作成
• Node.js ベースイメージの選択（node:18-alpine 推奨）
• 依存パッケージのインストール
• ビルドプロセスの設定
• エントリーポイント（Cloud Functions エミュレータまたは本番実行）の定義
FROM node:18-alpine WORKDIR /app COPY package*.json ./ RUN npm install COPY . . EXPOSE 5000 CMD ["npm", "run", "dev"]
3
.dockerignore ファイル作成
• node_modules
• npm-debug.log
• .env（本番環境では Secret Manager を使用）
• .git, .gitignore
4
ローカル Docker ビルド & テスト
• docker build -t firebase-budget:latest .
• docker run で動作確認
• ログ出力の確認
• 初期エラーの修正
docker build -t firebase-budget:latest . docker run -e GOOGLE_APPLICATION_CREDENTIALS=/app/key.json \ -v /path/to/key.json:/app/key.json \ firebase-budget:latest
✅ Day 1 チェックポイント
• Dockerfile が正常にビルドできている
• ローカルで docker run がエラーなく実行できている
• コンテナ内から Firebase への接続テストが成功している
Day
2

docker-compose 設定 & 本番構成

マルチコンテナ構成と環境変数管理

6-8時間
1
docker-compose.yml 作成（開発環境）
• version: '3.8' 以上
• services セクション（メインアプリ定義）
• environment / env_file の設定
• volumes（永続化）の設定（オプション）
version: '3.8' services: firebase-app: build: . environment: GOOGLE_APPLICATION_CREDENTIALS: /app/key.json NODE_ENV: development volumes: - /path/to/key.json:/app/key.json:ro ports: - "5000:5000" restart: unless-stopped
2
本番用 docker-compose.yml 作成
• NODE_ENV=production 設定
• リソース制限（memory, cpus）
• ローリングアップデート設定
• ヘルスチェック（HEALTHCHECK）設定
services: firebase-app: build: . environment: NODE_ENV: production resources: limits: memory: 1G cpus: '0.5' healthcheck: test: ["CMD", "curl", "-f", "http://localhost:5000/health"] interval: 30s timeout: 10s retries: 3
3
.env.example 作成
• 本番環境で必要なすべての環境変数をリスト化
• デフォルト値や説明をコメントで記載
• ダミー値は記載しない（セキュリティのため）
4
ローカル本番環境シミュレーション
• docker-compose -f docker-compose.prod.yml up
• 負荷テスト（Apache Bench, wrk）
• メモリ使用量、CPU 使用率の確認
• ログ出力の確認
docker-compose -f docker-compose.prod.yml up -d sleep 5 curl http://localhost:5000/health
✅ Day 2 チェックポイント
• docker-compose up で本番環境が起動する
• 環境変数が正しく設定されている
• ヘルスチェック機能が動作している
• リソース制限が適切に設定されている
Day
3

本番デプロイメント & 設定

顧客環境への展開とセキュリティ設定

8-10時間
1
本番サーバーの環境準備
• Docker & Docker Compose のインストール確認
• Linux ファイアウォール設定（ポート 5000 の制限）
• SSH キーベースの認証設定
• ディスク容量確認（ログ、キャッシュ用に 20GB 以上推奨）
2
シークレット・認証情報の配置
• Firebase サービスアカウントキー（JSON）を安全に配置
• Freee API トークンを環境変数として登録
• ファイルパーミッション設定（600 または 400）
• .env ファイルのバージョン管理除外設定
# 本番サーバーで mkdir -p /opt/firebase-budget/secrets chmod 700 /opt/firebase-budget/secrets cp firebase-key.json /opt/firebase-budget/secrets/ chmod 400 /opt/firebase-budget/secrets/firebase-key.json
3
Docker イメージの本番サーバーへの展開
• GitHub / Docker Hub からのプル、またはローカルビルド
• イメージタグ付け（firebase-budget:v1.0.0 など）
• デジタル署名確認（セキュリティベストプラクティス）
docker pull your-registry/firebase-budget:v1.0.0 # または docker build -t firebase-budget:v1.0.0 .
4
docker-compose up による本番実行
• docker-compose -f docker-compose.prod.yml up -d
• ログの確認（docker-compose logs -f）
• ヘルスチェック確認
• 初期データの同期テスト
✅ Day 3 チェックポイント
• コンテナが本番サーバーで起動している
• Freee API との接続が確立されている
• Firebase からのデータ取得が正常に動作している
• ログが適切に出力されている

🎯 全工程を通じた重要ポイント

並行作業の可能性

開発チームが複数いる場合、以下の作業を並行化できます：

  • Day 1: Dockerfile 作成と同時に本番サーバー準備（別チーム）
  • Day 2-3: docker-compose 設定と運用ドキュメント作成（別チーム）
  • Day 4: テストと同時に FAQ 作成

リスク管理

⚠️ 本番環境への移行リスク
Day 4 の統合テストで問題が見つかった場合、スケジュール調整が必要になります。
リスク軽減： • Day 2 の時点で本番環境構築の準備を進める
• Day 3 で本番への「小規模」移行テストを先行実施

セキュリティチェックリスト

  • Firebase サービスアカウントキーが .gitignore に含まれているか
  • .env ファイルがバージョン管理外か
  • 本番サーバーのファイルパーミッション（600 以下）が正しいか
  • ネットワークファイアウォール設定が適切か
  • Docker イメージに機密情報が埋め込まれていないか

💡 Docker セットアップのベストプラクティス

イメージサイズの最適化

Alpine Linux ベースイメージ（node:18-alpine）を使用することで、イメージサイズを 200MB 程度に保つことができます。
これにより、デプロイ時間が短くなり、ディスク容量の効率化も実現できます。

マルチステージビルド

本番環境では不要なツール（テストフレームワークなど）を除外するため、マルチステージビルドの採用を検討してください。

ヘルスチェックの重要性

HEALTHCHECK を設定することで、Docker デーモンやオーケストレーション・ツール（Kubernetes など）が自動的に不正なコンテナを検出・再起動できます。

ログ戦略

コンテナ化されたアプリケーションは stdout/stderr へログを出力すべきです。
これにより、docker logs や Docker Compose のログ機能で一元管理できます。
