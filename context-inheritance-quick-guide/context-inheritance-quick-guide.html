📌 Section 1: 導入 - なぜコンテキスト管理が必要か

🤔 「コンテキストウィンドウ」とは何か？

Claude Code では、AI（Claude）が「前のやりとりを覚えておける容量」が決まっています。これをコンテキストウィンドウと呼びます。

イメージ：会話できる容量の目安

  • トークン数（単位）で測定（おおよそ英単語 1～2 語 = 1 トークン）
  • Claude の最新モデルは 200,000 トークンまで覚えられる
  • つまり、かなり長い会話履歴を保持できる

⚠️ なぜ「前のセッションが忘れられる」のか？

新しいセッションが始まると、AI には「前のセッション」の情報が一切ありません。

graph TD A["セッション 1
前のやりとり"]; B["セッション終了"]; C["セッション 2
新しい開始"]; D["❌ 前のセッションの
情報が消えている"]; A --> B --> C --> D; style A fill:#e3f2fd style D fill:#ffebee

結果：

  • 「この機能のアーキテクチャをどうするか」が決まった → 次セッションで「決まった内容」を忘れられる
  • 「前回なぜこの方針にしたのか」の理由が不明になる
  • 毎セッション、一から説明し直さねばならない ⏰ 時間ロス

💡 クイックガイドの目的：この「情報喪失」を防ぎ、セッション間でスムーズにコンテキストを引き継ぐための実践的な手法を学ぶ

❌ Section 2: 課題シーン - Web アプリの複数機能開発で何が起きたか

実例：複数機能開発時のコンテキスト喪失

「ユーザー認証機能」を完成させた後、別セッションで「プロフィール編集機能」を開発する場合を例に。

timeline title セッション間でのコンテキスト喪失（複数機能開発） section セッション 1 ユーザー認証機能を実装 : パスワードルール・バリデーション決定 : メール確認の仕様決定 : エラーハンドリング設計完了 section 情報喪失 ❌ セッション終了 : 新しいセッション開始 : （前の仕様が消える） section セッション 2 プロフィール編集機能を開発開始 : 「パスワード変更時のルールなんだっけ？」 : メール確認の流れを忘れてた : 設計の不整合が発生

❌ 問題ある状態（Before）

  • セッション 1 の「ユーザー認証機能の仕様書」がない
  • パスワードのバリデーションルール（8文字以上など）をメモしていない
  • メール確認フローの詳細が文書化されていない
  • エラーメッセージのフォーマットが決まったことを記録していない

セッション 2 での問題：

  • 「プロフィール編集でパスワード変更できるようにしよう」 → パスワードの仕様を忘れてた
  • 前回と異なるバリデーションを実装 → データの不整合
  • 後からバグを発見 → 修正に時間がかかる 😤

✅ 改善後（After）

  • CLAUDE.md に「完成した機能と仕様」を記載
  • メモリに「ユーザー認証仕様書」を作成（バリデーション・フロー・エラー処理）
  • Git コミット：feat: implement user authentication with email verification
    詳細に仕様を記載
  • セッション開始時に「前セッションで完成した機能と、今回実装する機能の関連性」を述べる

セッション 2 での成果：

  • プロフィール編集機能から「前回の認証仕様」をすぐに参照できる
  • 一貫性のあるバリデーション・エラー処理を実装
  • 設計の整合性が保たれ、効率的に進む 😊

🔧 Section 3: 解決策フレームワーク - 5つの引き継ぎ方法

コンテキスト引き継ぎを成功させるには、複数の層（レイヤー）で情報を保存・共有する必要があります。

⚠️ 重要：CLAUDE.md と Auto memory の圧縮時の動作

公式ドキュメントでは「CLAUDE.md と Auto memory は圧縮後に再注入される」と説明されていますが、 これは「ディスク上に保存されている現在の内容が再注入される」という意味です。

つまり：セッション中に CLAUDE.md やメモリを更新しても、圧縮前にディスクに書き込んでおかないと、 圧縮後は古い情報が戻ってきます。

✅ 解決策：これが「圧縮直前自動化パターン」の役割です。 セッション終了時に自動で CLAUDE.md・メモリ・Git に情報を書き込んでから圧縮させることで、 次セッションで確実に引き継ぐことができます。

優先度順の 5 つの引き継ぎ方法

📚 公式リファレンス：

Claude Code の公式メモリシステムは CLAUDE.md ファイル（ユーザー記述）と Auto memory（Claude 自動記録）の 2 つです。 本ガイドは、これらをベースに「コンテキスト継承」をより包括的にまとめています。

1️⃣ CLAUDE.md（最重要）

最優先

プロジェクトの「現在のステータス・次にすべきこと」を記載。セッション開始時に最初に読むべき。

内容例：

## 現在のステータス 設計フェーズ完了・実装準備中 次のタスク：ユーザー認証機能の実装

2️⃣ メモリシステム

高優先

プロジェクト進捗・決定事項・理由を記録。セッション間で参照。

例：budget_management_architecture_decision.md

3️⃣ セッション開始時プリアンブル

中優先

セッション開始時に「前セッションの状態」を AI に伝える。

例：「前回バックエンドを Node.js + Express で実装することを決定した。フロントエンド実装を続ける」

※ 公式ドキュメントには明示的な記載がないユーザーの工夫です。CLAUDE.md や Auto memory と異なり、毎セッション手動で説明する必要があります。

4️⃣ Git コミット

中優先

重要な決定・マイルストーンをコミットメッセージに記録。

feat(auth): implement user authentication with JWT

※ 公式では operational な機能として説明（Claude が自動的に git 履歴を読んでコンテキストを理解）。メモリシステムとしての位置付けはコミュニティの実践的なアプローチです。

5️⃣ ドキュメント（README/DESIGN）

低優先

詳細な設計書・説明書をプロジェクト内に保存。

例：ARCHITECTURE.md, DESIGN_DECISIONS.md

📊 役割分担表：何を、どこに、いつ保存するか

5つの方法は、「保存する情報の粒度」「読み書きのタイミング」「セッション間での役割」が異なります。 それぞれが補完し合うことで、初めて「完全なコンテキスト引き継ぎ」が実現します。

方法 保存する情報 いつ読む いつ書く セッション間での役割
1️⃣ CLAUDE.md 「今ここ」を素早く把握

2️⃣ メモリ 「なぜそうした」を記録
決定事項・判断理由・懸念事項		 実装時に迷った時・判断の背景を確認する時 セッション中に手動記録（重要な決定時） 前の判断の理由を次セッションで思い出す
3️⃣ プリアンブル AI へのコンテキスト

💡 重要な違い：

  • CLAUDE.md： ファイル全体で「最新の状態」を示す（圧縮直前に常に更新）
  • メモリ： セッション間で蓄積される（追記型・削除しない）
  • プリアンブル： Claude が セッション開始時 に読むための「説明」
  • Git： コード変更の記録（ドキュメント変更ではなくコード履歴）
  • ドキュメント： セッション間で変わらない「設計仕様」（参考用）

🔒 ⚠️ セキュリティに関する重要な注意

CLAUDE.md・メモリ・コミットメッセージに記録してはいけない情報：

  • API キー・トークン・シークレット（.env で管理）
  • パスワード・パスフレーズ
  • クレジットカード・銀行口座情報
  • 顧客データ・個人情報（SSN、パスポート番号など）
  • 社外秘の設計・ビジネス戦略

✅ 正しい取扱い： これらはすべて .env・.gitignore・ローカル設定ファイルで管理し、Git にコミットしないでください。

📋 セットアップ手順：

# 1. .env ファイルを作成（機密情報はここに）
GOOGLE_API_KEY=your-api-key-here
OPENAI_API_KEY=your-api-key-here
DATABASE_PASSWORD=your-password

# 2. .gitignore に .env を追加
echo ".env" >> .gitignore
echo ".env.local" >> .gitignore

# 3. Git に追加しないことを確認
git status  # .env が表示されていないことを確認

# 4. CI/CD では環境変数を設定
# GitHub Actions の場合: Settings → Secrets → Add repository secret

実装方法（推奨：テンプレート型）

重要： シェルスクリプト内で「何を CLAUDE.md に書き込むか」を明示的に定義する必要があります。

✅ 初心者向けはテンプレート型を推奨します
シンプルで確実に動作します。複雑な自動化は次の「応用編」を参照してください。

⭐ 方法 1: テンプレート型（推奨）

初心者向け・推奨

CLAUDE.md に固定テンプレートを事前定義。圧縮直前にシェルスクリプトが日時を埋め込むだけ。

⚠️ テンプレートの内容・セクションをどう定義するか：
プロジェクト特性に応じて、CLAUDE.md のセクション構成を決めることが重要です。

ステップ 1: シェルスクリプト作成（3つの記録を自動実行）

ファイル: .claude/hooks/on-pre-compact.sh

⚠️ メモリ・Git コミット内容の決定方法
「何をメモリに記録するか」「コミットメッセージに何を含めるか」はプロジェクトごとに決定してください。一般的には、メモリには「決定の背景・理由」を、Git コミットには「実装内容」を記録します。 

#!/bin/bash
SESSION_DATA=$(cat)
SESSION_ID=$(echo "$SESSION_DATA" | jq -r '.session_id')
TIMESTAMP=$(date "+%Y-%m-%d %H:%M:%S")

# 1️⃣ CLAUDE.md に追記
cat >> CLAUDE.md << EOF

### Session: $SESSION_ID
- 実行日時：$TIMESTAMP
- 状態：圧縮実行
EOF

# 2️⃣ メモリに記録（.claude/memory/session-log.md）
mkdir -p .claude/memory
cat >> .claude/memory/session-log.md << EOF

## Session $SESSION_ID
- 日時：$TIMESTAMP
- ステータス：圧縮実行前に状態保存
EOF

# 3️⃣ Git コミット
git add -A
git commit -m "docs: auto-save session state before compression

Session: $SESSION_ID
Timestamp: $TIMESTAMP
Reason: PreCompact hook auto-execution" 2>/dev/null || true

exit 0

メリット： シンプルで確実に動作。テンプレートを定義すれば自動で更新。
デメリット： 「何を記録するか」は事前定義のため、融通がきかない。

💡 テンプレート型で十分な場合が多いです

開発初期段階・シンプルなプロジェクト向け。実装が簡単で確実に動作します。

📚 応用編：AI分析型（上級者向け）

セッション内容を Claude が自動分析して「重要な決定」を JSON で出力し、それを CLAUDE.md に自動記載する高度な方法です。テンプレート型では対応できない複雑なプロジェクト向けです。

⭐⭐⭐ 方法 3: AI分析型（上級向け）

中級者向け・応用編

Claude が会話を分析して「重要な決定」を JSON で出力 → シェルスクリプトが整形して CLAUDE.md に記載。

ステップ 1: Claude 分析スキルを作成

セッション圧縮直前に、Claude が会話を分析するプロンプト

⚠️ 「重要な決定」をどうやって判定するか？
Claude が自動分析するか、セッション開始時に「テーマ」を指定するか、など複数の方法があります 

## セッション分析プロンプト
本セッションの内容を JSON 形式で分析：
{
  "session_date": "日付",
  "key_decisions": ["決定事項1", "決定事項2"],
  "completed_tasks": ["完了タスク"],
  "pending_items": ["未完了項目"],
  "technical_notes": ["技術的メモ"]
}

形式で JSON だけを出力してください。

ステップ 2: シェルスクリプト（JSON を受け取り、3つの自動記録）

ファイル: .claude/hooks/on-pre-compact-ai.sh

#!/bin/bash
SESSION_DATA=$(cat)
SESSION_ID=$(echo "$SESSION_DATA" | jq -r '.session_id')
TIMESTAMP=$(date "+%Y-%m-%d %H:%M:%S")

# Claude からの分析 JSON を受け取る
ANALYSIS_JSON=$(echo "$SESSION_DATA" | jq -r '.analysis_json')

# JSON を解析
DECISIONS=$(echo "$ANALYSIS_JSON" | jq -r '.key_decisions[]')
COMPLETED=$(echo "$ANALYSIS_JSON" | jq -r '.completed_tasks[]')
PENDING=$(echo "$ANALYSIS_JSON" | jq -r '.pending_items[]')

# 1️⃣ CLAUDE.md に追記
cat >> CLAUDE.md << 'EOF'

## 🎯 現在のステータス（自動更新）
EOF

echo "- 更新時刻：$TIMESTAMP" >> CLAUDE.md
echo "- **完了したタスク：**" >> CLAUDE.md
echo "$COMPLETED" | sed 's/^/  - /' >> CLAUDE.md
echo "- **次のステップ：**" >> CLAUDE.md
echo "$PENDING" | sed 's/^/  - /' >> CLAUDE.md

# 2️⃣ メモリに詳細記録（.claude/memory/session-analysis.md）
mkdir -p .claude/memory
cat >> .claude/memory/session-analysis.md << EOF

## Session Analysis: $SESSION_ID
- 分析日時：$TIMESTAMP
- **重要な決定事項：**
EOF
echo "$DECISIONS" | sed 's/^/  - /' >> .claude/memory/session-analysis.md

# 3️⃣ Git コミット（JSON 形式で）
git add -A
COMMIT_MSG="docs: ai-analyzed session summary ($SESSION_ID)

Completed tasks:
$(echo "$COMPLETED" | sed 's/^/- /')

Next steps:
$(echo "$PENDING" | sed 's/^/- /')

Key decisions:
$(echo "$DECISIONS" | sed 's/^/- /')

Timestamp: $TIMESTAMP"

git commit -m "$COMMIT_MSG" 2>/dev/null || true

exit 0

メリット： 「重要な決定」を動的に抽出。会話の内容に応じて柔軟に対応。
デメリット： Claude API 呼び出しが必要で、実装がやや複雑。

⚠️ 応用編を検討する場合

方法3（AI分析型）は複雑な決定が多い大型プロジェクト・高度な自動化を目指す場合に適しています。ただし Claude API 呼び出しが必要で、実装がやや複雑です。

まずはテンプレート型で始めることをお勧めします。

❓ Section 5: よくある質問

Q: メモリを「圧縮」したら、古い内容は失われるのですか？

A: いいえ、失われません。Claude Code のメモリはマージ・統合しても、古い情報は MEMORY.md で参照可能です。ただし、古いメモリが「重要な決定」を含む場合は、CLAUDE.md に昇格させましょう。

例： 「GCP 版採用決定」 → メモリで管理 → セッションを跨いだら → CLAUDE.md に「現在のステータス：GCP 版採用済み」と書く

Q: CLAUDE.md が肥大化しないですか？

A: CLAUDE.md は「現在のステータス」だけに絞りましょう。詳細は別ファイル（ARCHITECTURE.md, DESIGN.md）に分離します。

CLAUDE.md の理想サイズ： 1-2 ページ（500-1000 字程度）

✅ CLAUDE.md に書く内容：

  • 現在のフェーズ（企画 / 設計 / 実装 / テスト / デプロイ）
  • このセッションの目標（3-5行）
  • 前回完了したタスク（3-5個）
  • 次のステップ（3-5個）
  • 重要な決定・方針（3-5個）

❌ CLAUDE.md に書かない内容：

  • 細かい実装の詳細 → ARCHITECTURE.md へ
  • セッション 1-10 の全履歴 → メモリ・別ファイルへ
  • エラーログ・デバッグ情報 → ローカルメモへ
  • 完了済みの古い決定 → メモリへ昇格、CLAUDE.md から削除

💡 肥大化を防ぐコツ： 毎セッション終了時に「古い情報を削除」することが重要です。CLAUDE.md は常に「次のセッション開始時に読むべき情報」だけを保持しましょう。

Q: セッション数が多いとき（10 回以上）はどうしますか？

A: 大型プロジェクトはマイルストーン単位でセッションサマリーを作成しましょう。

例： session_summary_2026-05-09.md に「セッション 1-3 の内容・決定事項・次の進め方」をまとめる

Q: 複数の AI（Claude + Gemini など）で開発する場合は？

A: CLAUDE.md は AI 非依存の「プロジェクト言語」で書きます。 AI の種類に関わらず読めるようにしましょう。

例： 「GCP 版を採用した」「Terraform で IaC 化する」（AI 種類を指定しない）

Q: コンテキストウィンドウが「満杯」になったら？

A: Claude Code が自動で「古い会話を削除」してくれます。ただし、セッション終了時に重要な決定をコミット・メモリに保存しておくことが重要です。

チェック： セッション終了前に「今日の決定はメモリ・コミット・CLAUDE.md に記録したか？」

Q: 「セッション開始時プリアンブル」は毎回必要ですか？

A: 短いプロジェクト： 不要（CLAUDE.md だけで十分）
長いプロジェクト・複雑な設計： 必須（1-2 文で前回の状態を述べる）

Q: 圧縮直前自動化で、メモリに「何を記録するか」はどうやって決めるのか？

A: 方法 1（テンプレート型）： CLAUDE.md のテンプレートで事前に「セクション」を定義。シェルスクリプトはそれに従うだけ。
方法 3（AI分析型）： Claude が会話を分析して「重要な決定」を抽出します。ただし、以下の 3つの実装方法があります：

  • A. セッション開始時にテーマ指定型：「今回のテーマ：ユーザー認証完成」と指定 → Claude がテーマに対する達成度を分析
  • B. 会話自動分析型：Claude が会話全体から「重要な決定」を自動抽出（判定基準は AI に依存）
  • C. 手動指定型：セッション終了時にユーザーが「重要な決定」を手動で指定

選択基準： プロジェクトの規模・セッション数・複雑度に応じて方法 1 または 3 を選択してください。

Q: Git コミットメッセージ自動化で、何を含めるべきですか？

A: 最低限以下を含めることをお勧めします：

  • セッション ID・実行日時
  • 完了したタスク（簡潔に）
  • 次のステップ・未完了項目

役割分担： 一般的には、メモリには「決定の背景・理由」を、Git コミットには「実装内容」を記録します。
