1. サブエージェントとは？

サブエージェントは、Claude Code内で特定のタスクに特化して機能するAIアシスタントです。メインの会話から独立したコンテキストや設定を持つことで、より効率的で安全な開発ワークフローを実現します。

💡 なぜサブエージェントを使うのか？ メインの会話にすべての情報を詰め込むと、コンテキストウィンドウを消費し、AIの焦点がぼやけ推論の質が落ちてしまいます。サブエージェントに専門タスクを委譲（Delegate）することで、プロジェクト全体の見通しを良くし、コストも最適化されます。

どうやってサブエージェントを起動するのか？

  • ① @メンションでの明示的な呼び出し (おすすめ)
    @Explore src/componentsディレクトリの内容を調査して
    @code-reviewer このプルリクエストの変更点をレビューして
  • ② 自然言語による指示
    「サブエージェントに任せて」「テストをバックグラウンドで回して」などの言葉で委譲。
  • ③ オプションを使ったコマンド起動
    claude --agent code-reviewer

2. 現場での実践ユースケース

⚠️ よくある誤解：「機能」ではなく「役割」で分ける 「メイン機能はメインエージェントに、サブ機能はサブエージェントに」ではありません！
同じソースコードを同時並行で書き換えると競合してしまいます。現場のベストプラクティスは、「実装担当」「テスト担当」「レビュアー」のように開発プロセスにおける役割（Role）で分担することです。

開発フローにおけるバトンパス（時系列）

  • 【開発前】 調査と設計（安全な下調べ）
    メインがコードを書く前に、@Explore に「既存コードへの影響範囲を調べてきて」と偵察に出す（読み取り専用で安全）。
  • 【開発中】 並行作業と専門アドバイス
    メイン画面でフロントエンドを実装している間、裏で @db-expert にSQLのチューニング案をバックグラウンドで考えさせておく（待ち時間ゼロ）。
  • 【開発後】 レビューとデバッグ（品質担保）
    実装が終わったら @code-reviewer を呼んで批判的な目線でチェックさせる。エラーが出たら @debugger にスタックトレースを解析させる。

3. Appendix: サブエージェントと「スキル」の関係性

サブエージェントはAG（別アーキテクチャ）で言うところの「特定のタスクをこなすための専用ワークフロー」を事前定義しておいたものです。

では、似た機能である「スキル（Skills）」とは何が違うのでしょうか？

比較項目 サブエージェント (Sub-agents) スキル (Skills)
概念の比喩 専門の担当者（職人） 手引書・手順書・能力
何をするか 自律的に考え、ツールを選び、タスクを実行 「この手順に従ってね」というプロジェクト固有の知識
設定の中身 使うモデル、ツールの利用制限、権限 特定のプロンプトテキスト、自動化スクリプトなど

実社会の会社組織に例えると…

サブエージェントは「担当者（ロール）」、スキルは「作業マニュアル・能力」と捉えると分かりやすいです。スキルは特定のエージェントに縛られず、独立して使い回せるのが最大の強みです。

  • 👩‍💼 経理エージェント (Sub-agent)
    • 📖 仕訳入力スキル (Skill)
    • 📖 決算処理スキル (Skill)
    • 📖 ビジネスマナースキル (共通Skill)
  • 👨‍💼 営業エージェント (Sub-agent)
    • 📖 商談・交渉スキル (Skill)
    • 📖 見積書作成スキル (Skill)
    • 📖 ビジネスマナースキル (共通Skill)
💡 強力な連携：サブエージェントにスキルを持たせる カスタムサブエージェントの Markdown フロントマター（skills: プロパティ）を使えば、そのエージェントにだけ特定のスキル（知識）をプリロードできます。無駄なコンテキストを消費せず、スマートな分業が可能です。

4. 実践編: CLAUDE.md を使った自動ルーティング

毎回ユーザーが @Explore や @code-reviewer と手動で呼び出すのは手間です。
プロジェクトのルートにある CLAUDE.md（メインエージェントのシステムプロンプト）に委譲のルールを書いておくことで、完全なオートメーション化が実現します。

CLAUDE.md の設定例（自動委譲の設計）

# Agent Delegation Rules (サブエージェント委譲ルール)

このプロジェクトでは役割分担を徹底しています。以下のタスクが発生した場合は、
あなた（メインエージェント）が直接処理するのではなく、
必ず適切なサブエージェントを呼び出して委譲（Delegate）してください：

1. **実装前の調査・影響範囲の確認**
   - コードを変更する前に、必ず `@Explore` にタスクを委譲すること。
   - レポートを受け取った後、あなた自身で実装を行うこと。

2. **コードのレビュー**
   - 機能の実装が完了したら、必ず `@code-reviewer` を呼び出すこと。
   - レビュアーの指摘事項を受け取り、修正が必要であればユーザーに提示すること。

3. **テストの実行**
   - コードを書き換えた後は、ユーザーの指示を待たずに `@test-runner` をバックグラウンドで起動し、並行してテストを回しておくこと。
🚀 Vibe Coding の極致へ この仕組みを作れば、ユーザーは「ログイン機能を追加して」と一行投げるだけでよくなります。 裏側では [メインが@Exploreに調査依頼] -> [メインが実装] -> [メインが@code-reviewerに自己評価を依頼] というタスクのバケツリレーが自動的かつ連鎖的に進行します。まるで自律した開発チームを指揮しているような体験が得られます。
